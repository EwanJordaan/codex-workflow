## Research baseline

The baseline is Anthropic's current Dynamic Workflows implementation, documented for Claude Code 2.1.154 and later:

- [Dynamic workflows](https://code.claude.com/docs/en/workflows)
- [TypeScript Agent SDK Workflow tool](https://code.claude.com/docs/en/agent-sdk/typescript#workflow)
- [Claude Code subagents](https://code.claude.com/docs/en/sub-agents)

Observed baseline features:

1. A workflow is a JavaScript orchestration script generated by Claude and executed outside the conversation. The script holds branching, loops, and intermediate results; only a consolidated result returns to the conversation.
2. Saved project workflows live in `.claude/workflows/`; user workflows live in `~/.claude/workflows/`. Project lookup walks from the working directory toward the repository root, and the nearest same-name workflow wins.
3. A saved file begins with `export const meta = { name, description }`, may declare phases, uses top-level `await`, reads structured input from global `args`, and orchestrates through `agent`, `parallel`, `pipeline`, and `phase`.
4. The public `Workflow` tool accepts inline `script`, saved `name`, `scriptPath`, structured `args`, or `resumeFromRunId`. It returns immediately with background task/run identifiers and script/transcript paths; syntax errors are reported in an `error` field.
5. Runs are background and inspectable. The `/workflows` view reports phases, agents, status, tokens, and elapsed time and supports pause/resume, stop, restart, and save.
6. Claude requests launch consent based on the active permission mode. Workflow agents use edit-accepting permissions while other capabilities still follow allowlists.
7. The runtime has no direct filesystem or shell access, no mid-run user input, at most 16 concurrent agents, and at most 1,000 agents per run.
8. Paused runs can resume in the same session. Completed unchanged `agent()` calls are cached. Restarting Claude starts the workflow fresh.
9. Claude warns about large projected runs and offers configurable size guidance. Workflows can route stages to different models.
10. Claude includes a bundled `/deep-research` workflow and can generate one-off workflows from natural-language opt-in or its `ultracode` effort mode.

## User stories

The MVP must let a Codex agent:

1. Discover a skill that explains when workflows help, the file contract, all MVP helpers, and exact create/run/wait steps.
2. Write a project-owned TypeScript workflow below `.codex/workflows/`.
3. Launch that file through a model-callable `run_workflow` tool with structured input.
4. Fan out bounded subagents, wait for their final messages, keep intermediate results in V8 variables, and return one consolidated serializable result.
5. Continue waiting through a model-callable `wait_workflow` tool when a run exceeds the initial yield window.
6. Receive actionable errors for unsafe paths, malformed metadata, unsupported syntax, agent failures, and limit violations.

## Functional requirements

### Skill

- Project skill path: `.codex/skills/codex-workflows/SKILL.md`.
- Trigger metadata must mention creating, inspecting, and running `.codex/workflows` scripts and multi-agent orchestration.
- The guide must be procedural and link to a complete runtime reference and this research/specification.
- `agents/openai.yaml` must contain matching display metadata and a default prompt that names `$codex-workflows`.

### Workflow discovery and validation

- Accept a project-relative `path` only.
- Resolve against `<cwd>/.codex/workflows`, canonicalize the root and file, and reject escape through `..` or symlink.
- Accept `.ts` and `.js`; cap source bytes at 128 KiB.
- Require a leading literal `export const meta = { ... }` declaration whose object contains non-empty string `name` and `description`.
- Preserve a TypeScript authoring surface while documenting that the MVP executable subset is JavaScript-compatible TypeScript.

### Execution

- Reuse the existing isolated V8 code-mode runtime; do not grant direct filesystem, process, network, console, or import access.
- Inject validated `meta`, structured `args`, and the four helper functions.
- Wrap the body in an async function so top-level `await` and `return` work.
- Serialize the final value into the tool output with workflow metadata.
- Yield after a configurable initial interval and support later waits/termination by cell ID.

### Agent orchestration

- Implement `agent(prompt, options)`, returning the final assistant message or parsed structured output.
- Implement ordered, bounded `pipeline`, callback-based `parallel`, and named `phase`.
- Reuse Codex's existing agent-control implementation through private nested-only spawn/wait tools. Do not expose legacy helper tools directly to the model.
- Respect existing session model, permissions, environment selections, depth limits, and agent execution limiter.
- Cap a workflow at 64 agents and helper concurrency at 4 for the MVP.

### Tool exposure

- Add `NestedOnly` to the tool exposure model: registered for dispatch and callable from code mode, but absent from the direct model tool list and code-mode prompt.
- Expose `run_workflow` and `wait_workflow` directly to the model only when collaboration tools are available.
- Keep `run_workflow` out of its own nested surface to prevent recursive launches.
- Start the lightweight code-mode dispatch worker for all turns; V8 itself remains lazy and is created only when a code cell executes.

## Runtime and security model

Workflow source is trusted project code but still runs in the same constrained V8 environment as code mode. The public launcher narrows executable files to `.codex/workflows`, bounds source size and spawned agents, and rejects unsupported metadata before V8 evaluation. Agent actions continue through ordinary Codex tool routing, hooks, permission checks, tracing, and environment selection.

The workflow script cannot bypass agent permissions because it only receives private spawn/wait capabilities. Spawned agents can use the capabilities already granted to the current session. Users and agents therefore must review a workflow before launch when it can prompt agents to edit or execute commands.

## Public API

### `run_workflow`

Input:

- `path: string` required, relative to the current working directory and below `.codex/workflows`.
- `args?: unknown` structured JSON exposed as global `args`.
- `yield_time_ms?: number` initial wait, default 1,000 ms.
- `max_output_tokens?: number` final/output budget.

Output is the ordinary code-cell output. A yielded result includes a cell ID. A completed result contains JSON with `meta` and `result`.

### `wait_workflow`

Input:

- `cell_id: string` required.
- `yield_time_ms?: number` default 30,000 ms.
- `max_output_tokens?: number` output budget.
- `terminate?: boolean` stop the runtime cell.

Output contains newly emitted content or the terminal result.

## Compatibility matrix

| Capability | Claude Code baseline | Codex MVP |
| --- | --- | --- |
| Script location | Project and user | Project `.codex/workflows` |
| Source | JavaScript | JS-compatible TypeScript or JavaScript |
| `meta`, `args`, top-level await/return | Yes | Yes |
| `agent`, `parallel`, `pipeline`, `phase` | Yes | Yes |
| Structured agent result | Yes | Supported JSON-Schema subset |
| Background execution | Yes | Yielded V8 cell plus `wait_workflow` |
| Phase/agent progress UI | Yes | No |
| Pause/resume and completed-call cache | Same session | Cell wait/terminate only; no cache |
| Concurrency/total cap | 16 / 1,000 | 4 / 64 |
| Saved command autocomplete | Yes | Skill-directed file launch |
| Consent UI and remembered approval | Yes | Existing Codex tool permissions |
| Model routing | Per stage | Per `agent` call where overrides are allowed |
| Built-in workflow | Deep research | None |
| Natural-language auto generation | Ultracode | Skill-triggered guidance |

## Validation plan

1. Unit-test metadata extraction, path containment, source wrapping, args injection, helper definition, size/extension rejection, and nested-only exposure.
2. Code-mode runtime test the generated wrapper with no agents to prove top-level await/return and output serialization.
3. Core integration-test a parent model launching a workflow whose hidden spawn/wait calls create a mocked subagent, collect its completed message, and return it.
4. Validate the skill with `quick_validate.py` and inspect its generated interface metadata.
5. Run repository formatting, scoped lint fixes, and the `codex-core` test target.
6. Run two independent reviewers over the complete diff and address actionable findings.
